# Design Principles — 设计规范生成规则

AI 生成 Design Tokens 时必须遵守以下推导流程，不得随机取值。本文件只描述推导规则；最终交付物中必须生成具体 CSS Variables 供组件和页面使用。

---

## 一、色彩

色彩生成不是先套主色色阶，而是先建立色盘。AI 必须结合产品气质、用户任务、目标平台、设计参考和色彩学关系来配色，再把色盘转成可用的 CSS Variables。

### 色盘生成流程

1. 明确色彩意图：判断产品需要传达信任、效率、温暖、活力、专业、安静、科技、医疗、生活方式等哪种气质。
2. 选择色相关系：根据产品气质选择单色、邻近色、互补色、分裂互补、三角色或中性色加强的色盘关系。
3. 构建基础色盘：包含品牌/主色、辅助强调色、中性色、语义色、背景表面色；如页面包含图表，再补充数据可视化色盘。
4. 调整色彩比例：大面积使用中性色和背景色，主色用于关键操作和当前状态，强调色用于少量高价值提示，语义色只表达真实状态。
5. 生成 token：把色盘扩展成可落地的变量，而不是机械地把一个主色拉成明暗梯度。

### 色彩判断方式

不要把颜色和行业、情绪做一一对应。色彩含义来自色相、饱和度、明度、面积比例、相邻颜色、品牌语境和使用位置的共同作用。同一个蓝色可能显得冷静、廉价、科技、医疗或忧郁；同一个橙色也可能显得温暖、警示、促销或幼稚。

AI 配色时应先回答这些问题：

1. 这个产品希望用户感到什么，而不是它属于哪个行业？
2. 界面需要安静、兴奋、可信、轻松、紧迫、精密还是亲近？
3. 哪些颜色适合大面积停留，哪些颜色只适合瞬间吸引注意？
4. 主色和辅助色之间是要形成和谐、张力、节奏还是方向感？
5. 当前页面的内容密度高不高，颜色应该帮助扫读还是制造情绪？
6. 色彩是否会和语义状态混淆，例如品牌红与错误红、品牌绿与成功绿？

使用色彩学时只做关系判断，不做模板判断：

- 色相用于建立色彩性格，但不能单独决定产品气质。
- 饱和度用于控制能量和商业感，面积越大越要谨慎。
- 明度用于建立可读性、空间层次和注意力停顿。
- 冷暖关系用于调整距离感和亲近感，不用于给行业贴标签。
- 互补、邻近、分裂互补、三角色等关系用于组织色盘结构，不直接决定最终风格。
- 中性色应有明确色温和明度秩序，而不是随手选择灰色。
- 语义色先保证状态识别，再与整体色盘调和。

### 色盘结构

最终色盘至少包含以下角色：

| 角色 | 生成方式 | 用途 |
|------|----------|------|
| Brand / Primary | 来自用户品牌色、设计参考或 AI 根据产品气质推荐 | 主操作、当前状态、关键路径 |
| Accent | 与主色形成可解释的色彩关系 | 少量提示、亮点、特殊入口 |
| Neutral | 带色温的灰阶，不得是随机纯灰 | 文本、边框、背景、分割 |
| Semantic | success、warning、error、info，各自语义清晰 | 状态反馈、风险、完成、提示 |
| Surface | 从 Neutral 和 Brand 微调得到 | 页面底色、卡片、浮层 |
| Data | 仅在图表或多分类数据中生成 | 可视化分类，避免与语义色混淆 |

### 色阶生成规则

- 色阶应从完整色盘推导，优先保持感知均匀，而不是只线性调整 HSL 明度。
- 若使用 HSL，应根据色相对饱和度和明度做人工修正；黄色、蓝色、紫色等不同色相的可读明度并不相同。
- 若环境支持 OKLCH / LCH 思路，应优先用感知更均匀的方式推导明暗层级。
- primary-500 必须是可用于主按钮和关键操作的稳定颜色；若原始品牌色过亮、过灰或过暗，需生成可交互版本，同时保留品牌气质。
- 浅色阶用于背景、hover、弱状态和轻提示；深色阶用于文字、pressed、深底强调和高对比状态。

### 渐变色彩路径

渐变不是把两个颜色混在一起，而是一条有语义、有节奏、有材质逻辑的色彩路径。AI 只有在渐变能明显服务品牌气质、空间层次、状态反馈或视觉焦点时才使用渐变；否则优先使用稳定纯色、背景层级、边框或阴影。

生成渐变前先回答：

1. 这个渐变承担什么角色：品牌、主操作、状态、装饰、数据编码，还是空间光感？
2. 两端颜色是否来自同一色盘关系，还是临时拼接的两个颜色？
3. 色相路径是否自然，中间过渡会不会发灰、发脏或产生泥色？
4. 明度路径是否稳定，是否会出现局部塌陷、脏暗块或刺眼高光？
5. 饱和度路径是否有控制，是否从高能量突然掉到灰脏区域？
6. 渐变所在控件是否仍然清楚可点击、可阅读、可识别状态？

渐变使用规则：

- 小尺寸控件和主按钮不适合承担复杂跨色相渐变；若使用渐变，应优先是同色相或邻近色的轻微变化。
- 跨冷暖、跨互补或跨高低明度的渐变必须有明确语义和过渡路径，不得直接把两个端点硬混。
- 渐变中间色必须被检查；如果中段看起来发灰、发脏、像污渍或材质不明，应重新选色或取消渐变。
- 渐变不能破坏文字对比度。按钮文字、图标和边界在渐变所有区域上都必须可读。
- 渐变不能替代状态语义。成功、警告、错误、禁用等状态仍需清晰文字、图标或形状辅助。
- 同一界面中渐变应少量使用；多个渐变同时出现时，必须有统一的色盘逻辑和光源逻辑。
- 渐变若只是为了“看起来高级”，但没有服务注意力、语义或品牌，应删除。

渐变 token 仅在实际使用时生成，推荐命名为语义角色，例如 `--gradient-primary-action`、`--gradient-brand-surface`、`--gradient-data-highlight`。不要生成没有使用场景的渐变 token。

判断示例：

- 合理渐变：来自同一色盘关系，色相路径、明度路径和饱和度路径都能解释；例如主按钮只做同色或邻近色的轻微深浅变化，用来表达可点击表面和 pressed 层次。
- 应拒绝的渐变：临时把两个无关系端点硬接在一起，中间发灰发脏，且说不清它服务品牌、状态、空间还是数据；例如小按钮上出现橙棕到灰绿的泥色过渡。

### 背景、文本与可访问性

- 背景层级从色盘中的 Neutral / Surface 推导，允许带轻微品牌色温，但不能抢内容注意力。
- 文本色从 Neutral 推导，必须形成 primary、secondary、tertiary、inverse 的清晰可读层级。
- 正文文字与背景对比度必须满足 WCAG AA；大文字、按钮文字、状态文字也必须通过可读性检查。
- 禁止把很浅的色阶直接当文字色；禁止用低对比彩色文字承载关键信息。
- 色彩不能成为唯一信息通道，状态必须同时通过文字、图标、形状或位置辅助识别。

### 推荐色彩变量命名

最终 CSS Variables 应覆盖本设计实际使用的色彩角色，并优先使用以下语义命名。若某类颜色没有被页面、组件或图表使用，不要为了凑齐列表生成空变量。

```css
--color-primary-100 到 --color-primary-900
--color-success-100 / --color-success-500
--color-warning-100 / --color-warning-500
--color-error-100 / --color-error-500
--color-info-100 / --color-info-500
--bg-page / --bg-surface / --bg-elevated
--text-primary / --text-secondary / --text-tertiary / --text-inverse
--gradient-*（仅在实际使用渐变时生成）
```

## 二、字体

字体 Token 不得照搬固定表，必须根据平台、信息密度、阅读任务和注意力权重生成。

### 字族规则

- 优先使用系统字体栈，中文界面必须优先使用平台中文系统字体。
- 禁止默认引入在线字体或远程字体文件。
- 若用户明确要求某个品牌字体，可使用该字体名作为首选，并保留系统字体作为 fallback。

### 字体注意力三轴

字体注意力由三个相互独立的维度共同决定，不能把它们绑定成一条“黑大粗到灰小细”的线性梯度。

| 维度 | 主要作用 | 设计判断 |
|------|----------|----------|
| 字号 / 行高 | 表达结构层级、阅读距离、内容分组 | 这个文本属于页面标题、区块标题、正文、元信息，还是局部标签？ |
| 字重 | 表达局部强调、扫描停顿、当前状态 | 这个文本是否需要在同字号内比相邻文本更快被扫到？ |
| 颜色 / 对比 | 表达语义状态、可交互性、弱化或强调 | 这个文本是主信息、辅助说明、禁用状态、风险状态，还是可点击入口？ |

使用规则：

- 字号、字重、颜色必须可以独立组合；不要让 `h1` 自动等于最大、最黑、最粗。
- 同一个字号可以有不同字重和颜色，例如普通正文、强调正文、次要正文、风险正文。
- 小字号也可以是高注意力信息，例如异常标签、倒计时、关键状态；它可以通过颜色、字重、位置或图标获得注意力。
- 大字号也可以保持低噪声，例如页面框架标题、月份标题、背景性章节名；它不一定需要最高字重或最高对比。
- 字重用于制造停顿，不是层级越高越粗；连续多个粗体会让页面失去节奏。
- 颜色用于表达语义和注意力状态，不是字号越小就越灰。
- Design System 展示字体时，应展示三轴组合样例，而不只是从 H1 到 Caption 的单列阶梯。

### 字号层级决策

先判断界面的阅读任务，再决定需要多少字号层级，不要为了凑齐表格而生成用不到的字号。

生成前先回答：

1. 用户是在快速扫读、长时间阅读、录入数据，还是比较指标？
2. 页面是单一主内容，还是有多个并列区块和辅助面板？
3. 信息密度是高、中、低？标题和正文需要拉开多大距离？
4. 哪些文字承担注意力权重，哪些只是元信息？
5. 移动端或 Web 画布是否限制了可读字号和行高？
6. 哪些文本需要通过字重或颜色强调，而不是通过继续放大字号强调？

常用语义层级包括页面标题、区块标题、卡片标题、强调正文、默认正文、次级正文、说明文字和标签文字。实际输出时只生成本设计会使用的层级，并确保相邻层级差异可感知但不突兀。

### 字体生成约束

- 字号必须形成稳定阶梯，相邻层级差异要可感知但不突兀。
- 行高随字号变化：标题更紧，正文更舒展。
- 字号整体密度由阅读距离、信息数量和操作频率决定，不由产品标签决定。
- 字重只用于建立停顿和局部强调，不得整页大量使用粗体。
- 颜色、字号、字重不得同向机械递减；必须出现合理的交叉组合，证明三轴是独立调度的。

### 推荐字体变量命名

最终 CSS Variables 至少包含基础字体、正文、说明文字、行高和字重变量；其他字号按实际设计需要补充。推荐使用以下语义命名，不使用 `size-1`、`size-2` 这类缺乏语义的名称：

```css
--font-family-base
--font-body / --font-caption
--font-h1 / --font-h2 / --font-h3（按需）
--font-body-lg / --font-body-sm / --font-label（按需）
--line-heading / --line-body / --line-caption
--font-weight-regular / --font-weight-medium / --font-weight-semibold / --font-weight-bold
```

字体 token 只描述字体尺寸、行高和字重，不得把颜色写进字体 token。文字颜色必须引用色彩 token，例如 `--text-primary`、`--text-secondary`、`--color-error-500`，并根据语义和注意力权重组合使用。

## 三、间距

间距 Token 不得照搬固定列表，必须根据内容关系生成一套有规律的 spacing scale。

### 间距决策

先判断内容关系，再决定 spacing scale 的颗粒度。间距不是装饰，而是把“属于一起”和“应该分开”的信息关系表达出来。

生成前先回答：

1. 页面是高频工具、内容浏览、数据比较、表单录入，还是展示型体验？
2. 用户需要一次看到更多信息，还是需要更强的停顿和呼吸感？
3. 哪些内容属于同一组，哪些内容需要明显分区？
4. 组件内部留白和区块外部留白是否有清楚差异？
5. 目标平台的触控、鼠标、键盘操作是否要求不同密度？

spacing scale 必须有一致递进关系，但层级数量按设计复杂度决定。简单页面可以少一些，复杂产品可以更细；不要生成没有被页面或组件使用的间距 token。

### 使用规则

- 页面外边距使用中到大号 spacing。
- 组件内边距使用小到中号 spacing。
- 组件之间使用中号 spacing。
- 区块之间使用大号 spacing。
- 同一页面不得混用没有进入 scale 的临时数字。

### 间距变量命名

最终 CSS Variables 至少包含组件内距、组件间距和区块间距所需变量；页面级变量仅在页面布局需要时生成。推荐使用以下语义命名：

```css
--space-xs / --space-sm / --space-md / --space-lg / --space-xl
--space-xxs / --space-2xl（按需）
--space-page-x / --space-page-y（按需）
--layout-card-min（按需）
```

## 四、圆角

圆角 Token 必须按风格推导，不得固定为某几个死值。

### 生成规则

先确定形态语言，再决定圆角层级。圆角 token 不需要机械生成完整套，只需要覆盖实际组件用到的尺寸关系。

常用语义包括：小控件圆角、主要控件圆角、页面级容器圆角、圆形/胶囊圆角。若设计中没有头像、开关或胶囊控件，不必生成 `radius-full`。

### 圆角校准方式

不要用行业或平台名称直接决定圆角大小。AI 应观察产品希望呈现的形态语言：精密或亲和、轻盈或稳重、密集或舒展、工具感或内容感。

- 高频操作控件需要稳定、清晰、可预期，圆角不应干扰识别。
- 内容卡片的圆角应与间距、阴影、边框共同形成统一形态。
- 大圆角会增加柔和感和块面感，但在高密度界面中可能浪费空间。
- 小圆角会提高秩序感和效率感，但过小可能显得生硬。
- 同一套界面中，圆角差异应来自组件尺寸和层级，而不是随意变化。

### 圆角变量命名

最终 CSS Variables 至少包含主要控件圆角；其他变量按实际组件需要补充：

```css
--radius-md
--radius-sm / --radius-lg / --radius-full（按需）
```

## 五、阴影

阴影 Token 必须按空间关系和注意力权重生成，不得固定为某几个死值。

### 生成规则

先判断界面是否需要靠阴影表达空间层级。若页面需要高密度扫读或已经能通过边框、背景差和留白建立层级，应减少阴影；若存在浮层、拖拽、悬浮或模态关系，再生成更完整的阴影 scale。

常用语义包括：普通表面、悬浮控件、模态浮层。若设计没有浮层或弹窗，不必生成最高阴影层级。

### 使用规则

- 阴影必须随空间层级递增：模糊半径、Y 偏移、透明度逐级增加。
- 阴影强弱由空间关系和注意力权重决定，不由行业、平台或风格标签决定。
- 若界面已经通过背景、边框、留白建立层级，应减少阴影噪声。
- 暗色模式不依赖阴影建立层级，改用边框、描边或背景明度差。

### 层级变量命名

最终 CSS Variables 至少包含边框层级；阴影变量按实际空间层级需要补充：

```css
--border-subtle / --border-strong
--shadow-sm / --shadow-md / --shadow-lg（按需）
```

## 六、HTML 输出约束

- 所有颜色、字号、间距、圆角、阴影必须引用 CSS Variables，**禁止 magic number**
- 禁止用 `position: absolute` 构建页面主体布局（stack 组件除外）
- 禁止内联 `style` 属性，全部使用 class
- 移动端模拟器容器固定宽度 **390px**（iPhone 14 标准）
- 暗色模式：默认不实现，除非用户明确要求

## 七、注意力权重 Token 使用规则

生成 Token 后必须结合 [visual-hierarchy.md](./visual-hierarchy.md) 使用，先估算页面信息单元的注意力权重，再决定 Token 如何落到视觉强度上。

- 高权重区域优先获得更强位置、尺寸、对比和可读性。
- 低权重信息使用 secondary/tertiary 文本、弱背景和更低视觉噪声。
- 主色、粗体、大字号、强阴影和高饱和状态色都是注意力资源，必须按权重分配。
- 重要性接近的元素应被组织成共同视觉组；重要性不同的元素必须呈现明显强弱差。

## 八、设计参考对 Token 的影响规则

用户提供设计参考时，AI 不应按品牌、平台或行业名称套模板，而应观察参考图真正呈现出的视觉关系。不得直接复制参考颜色值。

### 观察维度

- 色彩比例：背景、中性色、主色、强调色分别占多少视觉面积。
- 对比方式：依靠明度差、饱和度差、冷暖差、边框、阴影还是留白建立层级。
- 情绪密度：整体是克制、活跃、亲和、锋利、轻盈、厚重还是沉稳。
- 组件密度：按钮、卡片、列表、导航的间距和尺寸是紧凑还是舒展。
- 形态语言：圆角、边框、阴影、图标线条、分割方式是否一致。
- 注意力路径：用户第一眼被哪里吸引，第二眼如何移动。
- 语义状态：成功、警告、错误、信息提示是否与品牌色保持区分。

### 转换原则

- 从参考中提取关系，不复制表面样式。
- 若多个参考冲突，优先保留与当前产品任务最相关的关系。
- 若参考很好看但不适合目标平台或信息密度，应保留气质，重新调整 token。
- 若参考只是品牌氛围图，不能直接当组件规范；需要重新推导可交互颜色、文本颜色和状态颜色。
- 若参考中某种颜色很醒目，先判断它承担的是品牌、行动、状态、装饰还是数据分类，再决定是否进入 token。

## 九、暗色模式 Token 规则（仅当用户要求时生成）

暗色模式下，不得简单反转亮色 Token。必须基于同一套色盘重新校准明度、饱和度、对比度和色温。

| 亮色 Token | 暗色规则 |
|-----------|---------|
| bg-page | 从 Neutral / Brand 色温推导低明度背景，保持安静且不纯黑 |
| bg-surface | 比页面背景更清晰，但不产生刺眼块面 |
| bg-elevated | 与 surface 拉开层级，可通过边框、描边或明度差表达 |
| text-primary | 在深色背景上保持高可读性，避免纯白大面积刺眼 |
| text-secondary | 明显弱于主文字，但仍满足正文可读性 |
| text-tertiary | 用于说明、占位和禁用态，不承载关键信息 |
| primary-500 | 在暗色背景上重新校准为可交互主色，必要时调整亮度和饱和度 |
| 语义色 | 保持语义识别，同时避免在暗底上过度发光或过饱和 |
| shadow | 暗色模式下弱化阴影，更多使用描边、背景差和透明层建立层级 |
