# Quality Checklist — 输出前执行检查

最终输出 React 项目前必须逐项检查。发现问题先修复，再交付。

注意力权重验证优先于普通代码卫生检查。若页面视觉重点不成立，即使目录、链接和变量都正确，也不能交付。

---

## 一、结构检查

- 项目结构、文件命名符合 [artifact-structure.md](./artifact-structure.md)。
- `package.json`、`vite.config.ts`、`tailwind.config.ts`、`tsconfig.json`、`index.html` 均已生成。
- `src/main.tsx` 包含 `BrowserRouter` 和 `createRoot`。
- `src/App.tsx` 注册了所有页面路由，根路径重定向到首页。
- 所有依赖通过 npm 安装，无外部 CDN、在线字体或远程资源。

---

## 二、文本与代码卫生

在最终项目内检查并修复：

- 未替换模板占位符：`{{`、`}}`。
- 内联样式：`style=`（禁止在组件和页面中使用）。
- 字母、数字、emoji 或几何块图标占位。
- 页面或组件中直接内联的 SVG 路径（应从 `src/components/icons/` 导入）。
- 魔法数字颜色值（应来自 Tailwind Token 类名）。

---

## 三、Design Tokens

- Token 推导符合 [design-principles.md](./design-principles.md)。
- 色彩来自完整色盘和色彩学关系，而不是机械主色色阶或固定 HSL 区间。
- 渐变若存在，必须来自明确色彩路径和语义用途；没有脏灰中段、泥色过渡、无意义跨色相混合或破坏按钮可读性的问题。
- 页面注意力权重分配符合 [visual-hierarchy.md](./visual-hierarchy.md)。
- 主样式文件包含页面和组件实际引用的 CSS Variables，没有缺失变量或无意义空变量。
- 页面、组件、根目录总览页都使用同一套 Token。
- 色彩对比度满足 WCAG AA。
- 字体的字号、字重、文字颜色/对比是独立调度的三轴，没有被绑定成“越大越黑越粗、越小越灰越细”的单调梯度。
- 字体、间距、圆角、阴影层级一致，没有局部 magic number。

---

## 四、图标资产

- 图标生成与引用符合 [icon-library.md](./icon-library.md)。
- 每个图标是独立 SVG 文件，通过 `import X from '*.svg?react'` 导入为 React 组件使用，无内联 SVG 路径。
- 图标颜色通过 Tailwind 类名（`text-*`）控制，无硬编码颜色。
- 图标语义清晰，24x24 下可辨认，线宽、端点、转角和视觉重心一致。

---

## 五、组件资产

- 组件范围与变体符合 [component-catalog.md](./component-catalog.md)。
- 页面中出现的视觉样式能在 `src/components/ui/` 中找到来源，无临时自造样式。
- 弹窗是独立组件，由页面的 `useState` 控制，未在页面中内联弹窗结构。
- 数据卡片、图表、趋势、导航、控件均已作为独立组件存在。

---

## 六、页面蓝图

- 页面结构符合 [blueprint-schema.md](./blueprint-schema.md)。
- `type` 和 `control.variant` 没有自造值。
- 容器节点和叶子节点结构合法。
- Web 平台不使用移动端专属导航；使用 tab 时结构配套完整。

---

## 七、注意力权重验证

生成页面后，必须打开 Preview 或页面截图进行权重回读，不得只凭检查表判断。

### 权重回读

- 说出当前页面最先吸引注意的 1 到 3 个区域。
- 说明它们为什么应该获得更高注意力：任务相关、决策价值、行动可能性、时间敏感或状态严重。
- 说出哪些区域应保持低噪声：导航、元信息、辅助筛选、历史记录、说明文字等。
- 如果无法清楚回读，说明页面注意力设计不成立，必须修改。

### 视觉资源检查

- 高权重区域使用了哪些视觉通道：位置、尺寸、字号、对比、色彩、阴影、留白。
- 字体注意力是否同时检查了字号、字重、文字颜色/对比三个维度，而不是只看字号大小。
- 低权重区域是否降低了视觉噪声，避免强主色、粗体、大字号和强阴影。
- 重复列表项是否共享安静样式，仅当前项、异常项或最相关项提升权重。
- 元信息是否依附于主信息，没有独立占用强调资源。

---

## 八、视觉检查

- 阅读顺序清晰，视觉强度来自注意力权重分配，而不是静态套用层级规则。
- 能回读页面主要信息单元的注意力比例：高权重区域、常规权重区域、低权重区域各自合理。
- 没有“全是重点”的疲劳感；强主色、粗体大字、强阴影和高饱和标签按注意力预算分配。
- 没有为了装饰而在按钮、标签、小卡片上滥用渐变；按钮渐变若存在，文字、图标、边界和状态都必须清晰。
- 没有“黑大粗到灰小细”的字体单轴梯度；关键小字、安静大标题、同字号不同强调等组合应根据语义合理出现。
- 普通列表项、元信息、辅助卡片的视觉权重低于当前/关键内容，但仍保持可读。
- 通过眯眼测试和灰度测试后，页面重点仍然清晰。
- 首屏首先像一个真实产品界面，而不是说明文档或组件展示页。
- 领域关键结构完整：例如日历首页必须包含日期导航、时间轴或月/周视图、日程项、任务/提醒入口；不能只用通用卡片替代。
- 文本没有溢出、重叠、裁切或遮挡。
- 按钮、输入框、控件触控区域明显。
- 卡片、列表、导航之间间距一致。
- 页面内容像真实产品数据，不是重复模板文案。
- 移动端和 Web 画布尺寸符合目标平台。
- Web 预览不能因容器过窄而默认隐藏关键栏位，如侧边栏、主内容或右侧信息栏。
- 例如 Web Dashboard 若包含左侧导航、主内容和右侧信息栏，容器宽度应足以三栏并行显示。

---

## 九、交互验证

告知用户 `npm install && npm run dev` 后确认：

- 项目正常启动，浏览器可访问。
- 所有页面路由可通过 TabBar 或导航按钮跳转，无死链。
- 弹窗可正常打开和关闭。
- 页面布局符合目标平台：移动端设计应有合理最大宽度与居中或全宽策略，Web 设计无移动端断点侧漏。
- 关键组件、图标、导航栏在首屏可见，无布局溢出。
